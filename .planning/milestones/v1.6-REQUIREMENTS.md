## Archived Requirements

- [x] `REPLAY-01` — Seeku can execute replayable bounded-mission cases from stored prompt-plus-round fixtures instead of relying only on hand-written ad hoc mocks.
- [x] `REPLAY-02` — The replay set covers at least these mission shapes: converging shortlist, compare-ready, thin-but-stable, scattered clarification, retarget correction, and user stop/pause intent.
- [x] `REPLAY-03` — Each replay captures actual stop reason, mission phase, chat stop summary, and right-rail framing for comparison against expected posture.
- [x] `REPLAY-04` — Replay outputs classify mismatches into explicit failure buckets: `false_stop`, `late_stop`, `wrong_stage_report`, or `ui_semantic_mismatch`.
- [x] `GATE-01` — The milestone produces a replay-backed acceptance artifact with an explicit `go` or `hold` verdict for the current bounded mission scope.
- [x] `GATE-02` — Replay acceptance confirms that reportable mission output stays shortlist-first / compare-first and never defaults to premature top1 posture.
- [x] `GATE-03` — Replay acceptance confirms that mission corrections stay inside one mission object rather than spawning a parallel task.
- [x] `GATE-04` — Targeted web regression suites remain green after any replay-driven fixes.
- [x] `SCOPE-01` — The milestone remains focused on the shipped first mission type, `large-scope candidate search`, and does not introduce new mission categories.
- [x] `SCOPE-02` — The milestone does not add background continuation, multi-mission management, CRM workflow, or right-rail intervention controls.
