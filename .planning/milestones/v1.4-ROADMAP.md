# Milestone v1.4: Daily Deal Flow

**Status:** SHIPPED
**Archived:** 2026-04-17
**Phase range:** 09
**Total plans:** 4

## Overview

This milestone moved Seeku from a purely reactive search copilot toward a proactive opportunity product. On top of the shipped CLI runtime and visible panel foundation, Seeku now generates an in-product daily deal flow that models user direction, ranks candidates by directional fit, explains why to act now, and captures feedback that changes subsequent lists.

## Included Phase

### Phase 09: Daily Deal Flow

**Goal:** Add a proactive daily cofounder deal flow that ranks existing Seeku candidates by goal-direction match, explains why each person is worth contacting, and learns from explicit and implicit feedback.

**Plans:**
- [x] `09-01-PLAN.md` — Goal model and candidate direction foundation
- [x] `09-02-PLAN.md` — Opportunity scoring and daily curation pipeline
- [x] `09-03-PLAN.md` — Deal flow surface and feedback capture
- [x] `09-04-PLAN.md` — Drift logic, learning loop, and acceptance

## Delivered Outcome

- Added a reusable direction taxonomy, candidate direction profiling, and a user goal model in the shared search layer.
- Added an interpretable opportunity scorer and daily curator that keep direction match as the primary ranking signal.
- Added a dedicated `/deal-flow` web surface with:
  - top-three daily priorities
  - more-opportunities queue
  - `why this person`
  - `why now`
  - `how to approach`
  - confidence and uncertainty cues
- Added explicit feedback actions:
  - `感兴趣`
  - `不感兴趣`
  - `已联系`
  - `稍后再看`
- Added implicit interaction capture for:
  - detail opens
  - evidence expansion
  - dwell
  - repeat surfacing
- Added first-version drift handling so recent behavior can diverge from the explicit long-term goal without hiding that shift.

## Key Decisions

- Keep the proactive deal flow as an in-product surface rather than bundling external notifications or auto-outreach in the first milestone.
- Reuse the shipped search/runtime stack instead of introducing a new persistence-heavy ranking system.
- Keep candidate ranking direction-first; recency and engagement may tune order but may not rescue obvious mismatches.
- Use process-local viewer learning state first and defer durable persistence until the product contract settles.

## Deferred / Watch Items

- External delivery, outreach drafting, CRM, and corpus expansion remain deferred to later milestones.
- The deal-flow learning loop is still process-local and not yet durably persisted.
- `pnpm exec tsc -p apps/api/tsconfig.json --noEmit` is still blocked by the pre-existing `apps/api/src/routes/admin-claims.ts` type error outside this milestone's scope.
