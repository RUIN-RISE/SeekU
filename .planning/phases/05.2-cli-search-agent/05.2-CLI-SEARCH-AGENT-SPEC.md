# CLI Search Agent Spec

## Why This Exists

Seeku CLI 已经从“脚本式搜索命令”走到“会话式搜索助手”，但它还没有完全变成一个真正的搜寻智能体。

当前阶段的主要矛盾已经比较明确：

- 不是命令不够多
- 不是交互不够花哨
- 而是用户还没有持续感受到 Bonjour 搜索的核心价值

下一阶段的目标，不是继续堆零散功能，而是把 CLI 收敛成一个明确的产品形态：

**一个面向找人任务的 CLI 搜寻智能体。**

它的工作不是执行一次 `search`，而是陪用户完成完整的找人过程：

`表达目标 -> 收敛需求 -> 给出 shortlist -> 解释结果 -> 引导下一步动作`

## Product Thesis

Seeku CLI 的核心价值，不只是“搜到一些人”，而是：

1. 快速把模糊需求转成可执行搜索
2. 在宽条件下稳定给出一批可看的候选人
3. 明确告诉用户这些人为什么值得看
4. 让用户知道这些结果是不是来自 Bonjour、数据新不新
5. 给出清晰的下一步动作，例如打开 Bonjour Profile 深看

一句话定义：

**Seeku CLI Search Agent 是一个帮助用户在命令行里完成“找人、判断、继续跟进”的智能体，而不是单次检索工具。**

## Who It Is For

主要用户：

- 招聘方 / Sourcer
- 创业者 / Hiring Manager
- 技术负责人
- 任何需要快速从 Bonjour 数据里找到相关候选人的人

他们的真实诉求通常不是：

- “让我学会写精确的检索 query”

而是：

- “先给我一批靠谱的人”
- “告诉我为什么是这些人”
- “我下一步点谁”

## Core User Job

用户想完成的不是“运行一个命令”，而是一个连续决策任务：

1. 说出目标
2. 看到系统理解是否正确
3. 快速获得一批候选人
4. 判断哪些值得继续看
5. 打开 Bonjour 主页继续跟进

这意味着 CLI 必须服务下面这个任务节奏：

`clarify -> retrieve -> shortlist -> inspect -> open -> refine`

而不是：

`input -> output -> quit`

## What The Agent Must Do

### 1. Search Agent

职责：

- 理解自然语言需求
- 自动识别角色、技能、地点、经验、来源偏好
- 在用户表达模糊时主动收敛条件
- 在召回不足时主动放宽条件
- 保证第一屏尽量有结果

对用户来说的感知应该是：

- “它在帮我找人”
- 而不是
- “它在要求我写更标准的查询”

### 2. Decision Agent

职责：

- 生成 shortlist
- 解释为什么是这些人
- 展示来源和数据新鲜度
- 支持简单比较与继续 refine

对用户来说的感知应该是：

- “我知道为什么该点开这个人”

### 3. Action Agent

职责：

- 输出下一步动作
- 打开候选人的 Bonjour 链接
- 帮用户继续 refine
- 未来可扩展到导出、分享、后续跟进动作

对用户来说的感知应该是：

- “CLI 不只是告诉我结果，还告诉我下一步怎么做”

## MVP Definition

下一阶段的 MVP 不应该追求“大而全”，而应该先确保主价值链完整。

### MVP Goal

让用户在命令行里完成以下闭环：

`输入目标 -> 看到 shortlist -> 知道为什么 -> 知道数据来自 Bonjour 且有多新 -> 打开 Bonjour Profile`

### MVP Must-Haves

1. 宽查询稳定出人
   - 如 `杭州`
   - 如 `北京`
   - 如 `杭州 python`

2. shortlist 可判断
   - 候选人名字
   - 匹配原因
   - 来源
   - freshness
   - Bonjour 链接

3. detail 页可继续动作
   - why
   - refine
   - open bonjour

4. 第一屏不依赖 LLM 成功
   - LLM 是增强层
   - 不是门禁层

### MVP Explicit Non-Goals

以下内容暂时不应该成为下一阶段主目标：

- 复杂 compare pool 打磨
- 历史会话精细化管理
- 导出 / 分享 / 认领流程
- CLI 内直接展示微信等联系方式
- 复杂 agent orchestration

## UX Principles

### Principle 1: First Screen Must Be Useful

第一屏最重要的不是完美，而是有用。

如果用户输入宽查询，系统应该：

- 先给一批结果
- 再引导 refine

而不是：

- 先教育用户写更精确的 query

### Principle 2: Be Honest About Data

CLI 必须诚实显示：

- 来源是否明确
- 数据是否新鲜
- 哪些字段未知

不要伪装成“实时”或“完整”。

推荐展示：

- `Source: Bonjour`
- `Source: Mixed`
- `Source: Unknown`
- `Last synced: ...`
- `Latest evidence: ...`

### Principle 3: Link Out Instead Of Over-Embedding

CLI 不需要把所有联系信息都塞进来。

更合理的路径是：

- CLI 负责 shortlist 和决策
- Bonjour 页面负责进一步查看和联系

所以当前阶段只展示：

- `Bonjour Profile Link`

而不直接展示微信等敏感联系方式。

### Principle 4: LLM Enhances, Not Blocks

LLM 可以用于：

- query refinement
- why explanation
- candidate summary

但不能决定：

- 第一屏有没有结果

## User Journey

### Journey A: Broad Discovery

用户输入：

`找杭州的人`

系统行为：

1. 回显理解：地点是杭州，其他条件未限定
2. 直接给 shortlist
3. 每个人展示来源、新鲜度、Bonjour link
4. 用户选择一个打开 Bonjour

这是最基本的“先给我人看”的路径。

### Journey B: Structured Search

用户输入：

`找做 python 后端的工程师，在杭州`

系统行为：

1. 提取角色、技能、地点
2. 如果经验缺失，可继续补充
3. 给出 shortlist
4. 用户查看 why
5. 用户 refine 或打开 Bonjour

### Journey C: Guided Refinement

用户输入：

`给我几个 Bonjour 人选`

系统行为：

1. 不把 `bonjour` 当硬过滤导致 0 结果
2. 把它当来源偏好
3. 给出可看的 shortlist
4. 用户进一步 refine 条件

## What The Results Must Communicate

每个 shortlist item 至少要回答：

1. 这个人是谁
2. 为什么在这里
3. 数据从哪里来
4. 数据新不新
5. 去哪继续看

下一阶段推荐的 shortlist 信息结构：

- Name
- Match reason
- Source
- Last synced / latest evidence
- Bonjour profile link

推荐的 detail 信息结构：

- Profile summary
- Why matched
- Evidence sources
- Last synced
- Latest evidence
- Bonjour profile link

## Proposed Commands

保留已有命令式交互，但围绕“搜寻智能体”重新定义动作。

### Shortlist Level

- `v 2`
  查看第 2 位候选人

- `o 2`
  打开第 2 位候选人的 Bonjour 页面

- `why 2`
  查看第 2 位候选人的匹配依据

- `r`
  继续 refine

- `sort tech`
  按技术匹配排序

### Detail Level

- `open`
  打开 Bonjour 页面

- `why`
  展开评分依据

- `refine`
  基于当前候选人收窄或放宽搜索

- `back`
  返回 shortlist

## Data Requirements

为了让这个 Agent 真正成立，数据层至少要支撑这些字段：

### P0

- `facetLocation`
- `facetSource`
- `latestEvidenceAt`
- `lastSyncedAt`
- `bonjourProfileUrl`

### P1

- `primaryHeadline`
- 更可靠的 `facetRole`
- 更可靠的 `facetTags`

## Technical Implications

### 1. Retrieval Must Favor Recall For Broad Queries

宽查询时，系统要优先保证召回，而不是过早追求精确。

意味着：

- 地点-only 查询必须稳定出人
- sourceBias 应该是 soft preference
- fallback search 是产品必要能力，不是临时 hack

### 2. Rendering Layer Must Surface Value Signals

渲染层要把真正重要的信号露出来：

- 来源
- freshness
- Bonjour profile link

而不是只展示名字和分数。

### 3. Agent Memory Should Serve Search, Not Become A Feature Sink

记忆能力可以有，但目的应该是：

- 更顺畅地继续 refine

而不是把 CLI 变成一个充满边缘命令的状态机。

## Immediate Roadmap

### Phase 05.2

目标：

**让用户感知 Bonjour 的价值**

任务：

1. 在 shortlist 展示 source / freshness / bonjour link
2. 在 detail 展示 source / freshness / bonjour link
3. 增加 `open bonjour` 动作

### Phase 05.3

目标：

**提升数据质量，让 shortlist 更可判断**

任务：

1. headline 填充
2. facetSource 填充
3. 角色与标签质量提升

### Phase 05.4

目标：

**从搜索助手演进到更完整的搜寻智能体**

任务：

1. 更强的 compare
2. 导出 shortlist
3. 更自然的 refine / rerank

## Success Criteria

下一阶段是否成功，不看“命令多不多”，而看下面这些问题能不能回答“是”：

1. 用户输入 `杭州`，是不是总能拿到一批可看的候选人？
2. 用户是不是一眼就知道结果是否来自 Bonjour？
3. 用户是不是一眼就知道数据新不新？
4. 用户是不是能直接打开 Bonjour 页面继续动作？
5. 用户是不是不用学复杂命令，也能完成“找人 -> 判断 -> 打开 Profile”？

## One-Sentence Summary

**Seeku CLI 的下一阶段，不是继续做更复杂的命令行工具，而是把它做成一个真正帮助用户完成“找人、判断、打开 Bonjour 深看”的 CLI 搜寻智能体。**
