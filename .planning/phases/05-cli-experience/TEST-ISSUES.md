## 🛑 MAJOR: 严重缺陷 (2)

### 1. 无效输入传递漏洞 (ISSUE-001)
- **关联测试**: B1 (空输入提交), B4 (纯空格)
- **代码位置**: `apps/worker/src/cli/chat.ts:78`
- **复现步骤**: 启动交互 CLI -> 直接无视一直敲回车（或输入完全由空格组成的字符串）。
- **影响分析**: 交互层并没有对 `input.trim().length === 0` 做任何前置校验与拒绝（仅有一处粗略的针对追问回答的校验）。这意味着查询构建器最终拼凑了一个空白查询串提交给向量检索引擎（以及大模型），不仅浪费了接口调用 Quota Token，更有可能在底层产生全表捞取致使性能恶化。
- **修复方案**: 在 `askFollowUp` 以及 `index.ts` 中拦截一切无效字符串，若检测到纯空白/等效非法输入，强制重试或用系统指定的默认 Fallback 兜底词替代。

### 2. 打分引擎遗忘消费关键搜索意图 (ISSUE-002)
- **关联测试**: A8 (高级职位 / Senior 搜索)
- **代码位置**: `apps/worker/src/cli/scorer.ts:32` ( HybridScoringEngine 规则 )
- **复现步骤**: 搜索描述要求 “具有 5年以上经验的高级工程师”。
- **影响分析**: `chat.ts` 尽职尽责地向 `conditions` 吐出了 `experience: "5年以上"` 和 `role: "高级工程师"` 两个优质特征。然而由于评分引擎的设计断层，**目前的 Scoring 规则只消耗了 `skills` 和 `locations` 这两个键**。这种结构遗漏使得那些带有资历请求的查询在打分后无法精准提权对应的老练工程师，完全破获了该项自然语言调参特征在最终排行中的有效性。
- **修复方案**: 在 `calculateCareerStability` 以及其它相关评分 Rule 方法中引入对 `conditions.experience` 和 `conditions.role` 的对比/权重计算逻辑（例如结合候选人的工作年限或 Headline 中的职级）。

## 🟡 MINOR: 体验/轻微缺陷 (1)

### 3. 多轮追问产出冗余重叠数组 (ISSUE-003)
- **关联测试**: B5 (重复技能搜素)
- **代码位置**: `apps/worker/src/cli/chat.ts:106`
- **复现步骤**: 提问 "Java", 后续追问又回答 "Java"。
- **影响分析**: 当前条件对象的融合方式为粗劣的展开运算符 `[...(conditions.skills || []), ...(extra.skills || [])]`。该逻辑没有自带去重 (Deduplication) 逻辑，从而导致 SearchIntent 的 Skills 数组随着追问变得浮肿且包含多个重复同义词。
- **修复方案**: 改写为使用 `Array.from(new Set(...))` 去除纯字符重复项，提升向下层 API 派发查询时的整洁度和检索效率。
