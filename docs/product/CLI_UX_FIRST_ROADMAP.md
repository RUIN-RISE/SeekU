# Seeku CLI UX-First Upgrade Roadmap

## 1. 产品判断

Seeku 当前最有价值的产品形态不是“命令行工具本身”，而是“一个能把模糊招聘需求快速收敛成 shortlist 决策的搜寻智能体”。

从产品经理视角看，下一阶段不应该优先追求功能数量，而应该优先解决 4 个用户体验核心问题：

1. 可学性不足
   用户第一次进入 shortlist 后，不容易理解“下一步该按什么”“哪些键是热键，哪些是命令”。

2. 决策流不顺
   搜索、详情、compare、export 之间切换成本偏高，用户要记忆很多状态。

3. 系统反馈不稳定
   429、超时、后台任务、导出等行为在交互态里容易打断心流，降低信任感。

4. 交付形态不完整
   目前是开发者友好的 Node CLI，不是面向招聘/Founder/猎头的“可直接安装使用的产品”。

## 2. 当前版本与平台现状

基于当前仓库代码，现状如下：

- 当前不是独立的 “Windows 版” 和 “Mac 版” 两套产品。
- 当前是一套基于 Node.js 的跨平台 CLI。
- 运行方式本质上是统一的：
  - `npx tsx apps/worker/src/cli.ts ...`
  - 或构建后 `node apps/worker/dist/cli.js`
- 只有少量系统级行为做了平台分支，例如打开浏览器时区分：
  - macOS: `open`
  - Windows: `start`
  - Linux: `xdg-open`

版本上目前存在两层信息：

- 用户可见版本：`Seeku CLI v1.1.0`
- `package.json` 版本：`0.0.1`

这说明当前“产品版本”与“包版本”还没有统一，后续需要收口。

## 3. 产品目标

未来 2 个版本周期，Seeku CLI 的核心目标应该是：

1. 让第一次使用的用户在 3 分钟内完成一次成功 shortlist。
2. 让用户在不看文档的情况下完成详情查看、compare、导出。
3. 让失败场景可恢复，不打断当前思路。
4. 把它从“工程工具”升级为“可交付给非工程用户的招聘产品”。

## 4. UX 优先级原则

后续所有升级建议都应遵循这 6 条：

1. 先减少思考成本，再增加能力。
2. 先保证主路径顺滑，再补长尾功能。
3. 所有系统行为都要给出明确反馈和下一步建议。
4. 尽量让用户“看到”状态，而不是“记住”状态。
5. 所有高频操作都要支持一键完成。
6. Mac 与 Windows 只允许分发形态不同，不允许交互规则分叉。

## 5. 升级路线图

### Phase 1: 把主路径做顺

目标：让“搜索 -> shortlist -> detail -> compare -> export”形成无解释可完成的闭环。

优先项：

1. 统一交互模型
   - 固定热键语义：导航、动作、命令模式三层清晰分离
   - 所有视图统一 `?` 帮助、`q` 退出、`Esc/back` 返回
   - shortlist/detail/compare 的操作语言统一

2. 强化下一步引导
   - 每个视图底部只显示 3 到 5 个最值得执行的动作
   - compare 不满足人数时，直接提示“下一步怎么补人”
   - 导出成功后给出“下一步建议”，而不只是文件路径

3. 可见状态设计
   - 当前选中项、已入 compare pool、当前排序、当前筛选条件全部可见
   - refine 后要清晰展示“这轮发生了什么变化”
   - compare 输出明确标记“推荐先看谁、为什么、下一步点哪里”

4. 错误与等待体验
   - 429、超时、降级结果必须静默可控，不打断 prompt
   - 长操作都需要“正在做什么”的说明
   - 异常后提供恢复动作：重试、跳过、继续当前 shortlist

建议验收指标：

- 首次用户完成一次 shortlist 的成功率 > 80%
- shortlist 内高频操作不依赖帮助文档
- 失败日志不污染交互 prompt

### Phase 2: 把“搜到人”升级为“做决策”

目标：让 Seeku 不只是搜索器，而是决策工作台。

优先项：

1. Compare 工作台升级
   - 支持固定 compare 槽位
   - 支持“推荐结论 + 风险提示 + 下一步动作”
   - 支持从 shortlist 一键补人到 compare

2. 解释质量升级
   - 区分“为什么命中本次查询”与“候选人长期画像”
   - 解释结构固定化，避免每次风格漂移
   - 对“匹配但不推荐”的候选人明确给出原因

3. 导出体验升级
   - 提供 Founder 版、招聘经理版、猎头版导出模板
   - 导出内容默认包含建议动作，而不是只有字段罗列
   - 支持“导出 compare 结论摘要”

4. 会话连续性
   - 保存最近会话
   - 支持恢复上次 shortlist
   - 支持“基于上轮继续 refine”

建议验收指标：

- compare 使用率提升
- export 后回流率提升
- 用户完成 shortlist 到决策的时间缩短

### Phase 3: 把 CLI 做成真正可交付产品

目标：让非工程用户也能稳定安装和使用。

优先项：

1. 分发与安装
   - 提供 macOS 与 Windows 的独立发行物
   - 至少支持：
     - macOS universal app/binary
     - Windows x64 installer 或单文件可执行程序
   - 保留 Node 版作为开发者模式

2. 首次启动体验
   - 增加 `doctor`/环境诊断
   - 首次运行自动检查数据库、API key、网络连通性
   - 启动失败时输出面向用户的解决建议，而非底层异常

3. 版本策略统一
   - 产品版本统一为 `v1.x.x`
   - 不再出现“banner 是 1.1.0，但 package 是 0.0.1”的分裂状态
   - Mac 和 Windows 共享同一产品版本号，只区分构建产物

4. 可靠性建设
   - 关键路径埋点：搜索成功率、429 率、导出率、compare 使用率
   - 崩溃日志与匿名诊断
   - 回归测试覆盖交互主路径

建议验收指标：

- 非工程用户可在 10 分钟内完成安装与首搜
- 安装后首次运行成功率 > 90%
- 平台差异只体现在安装方式，不体现在交互规则

### Phase 4: 从“个人助手”升级为“团队工作流”

目标：让 Seeku 成为招聘协作入口，而不是单人工具。

优先项：

1. shortlist 分享与回看
2. 团队注释与标签
3. 候选人状态流转
4. 与外部 ATS / Notion / 飞书的导出或同步

这部分优先级低于前 3 个阶段，不建议现在先做。

## 6. 建议的版本规划

### v1.2

定义：交互一致性版本

必须完成：

- 统一热键与命令模式
- 帮助与提示体系重写
- compare pool 可见化
- 429 / 超时 / 后台任务不打断交互
- shortlist/detail/compare 主路径回归测试

### v1.3

定义：决策工作台版本

必须完成：

- compare 工作台增强
- 更强的 why matched / why not matched
- 会话恢复
- 模板化导出

### v1.4

定义：可分发版本

必须完成：

- macOS / Windows 打包产物
- 首次启动自检
- 统一版本体系
- 面向非工程用户的安装文档

## 7. 当前最值得先做的 10 个任务

1. 补 `Esc` 返回/取消，统一所有视图的退路。
2. detail / compare / shortlist 三个视图底部提示统一格式。
3. 增加 refine 草稿回显，告诉用户“这轮改了什么”。
4. compare 增加“为什么建议先看这个人”的固定结构。
5. export 成功后显示用途导向提示，而不是只有路径。
6. 加 `doctor` 命令，检查数据库、LLM key、Bonjour 可用性。
7. 把版本号统一成单一来源。
8. 增加会话保存与恢复。
9. 设计 macOS / Windows 的打包与安装方案。
10. 建立 UX 指标面板：首搜成功率、compare 使用率、导出率、429 率。

## 8. 关于平台版本的明确结论

一句话结论：

当前不是 Win 版和 Mac 版两个独立产品，而是一套跨平台 Node CLI；用户可见版本是 `v1.1.0`，但包版本仍是 `0.0.1`，这需要后续统一。
