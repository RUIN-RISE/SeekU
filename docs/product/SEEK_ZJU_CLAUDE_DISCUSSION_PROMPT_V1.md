# Seek ZJU 给 Claude Code 的讨论提示词

请把下面内容当作项目背景与当前方向，帮助我们评审 `Seek ZJU` 的项目方案，并提出更具体的产品与技术建议。

## 1. 项目背景

我们准备单独启动一个新项目，名称暂定为 `Seek ZJU`。

这个项目不是传统的招聘搜索产品，而是一个面向浙江大学 AI 高价值校友的情报与联络系统。

短期目标：

- 一期先聚焦中国国内
- 交付 100 位高质量浙江大学 AI 校友的结构化名单

长期目标：

- 沉淀为可持续更新的“浙江大学 AI 人才情报系统”

## 2. 一期目标人群

一期候选人必须尽量满足：

- 必须是浙江大学校友
- 当前主要在中国国内
- 属于狭义 AI 人才
- 优先在中国头部 AI / 大模型 / AI Infra / AI 研究相关企业或机构任职

这里的“AI 人才”优先指：

- 大模型算法工程师
- 机器学习工程师
- AI 科学家 / 研究员
- 多模态 / 视觉 / 推荐 / Agent / ML Infra 方向核心成员
- AI 技术负责人
- AI 投资人或 AI 产业研究型人才

一期暂不优先：

- Bay Area 与海外
- 泛互联网岗位
- 与 AI 关系较弱的运营、市场、销售岗位

## 3. 对外交付字段

一期计划交付一个表格，字段建议包括：

- 姓名
- 联系方式
- 性别
- 年龄
- 入学年份
- 专业
- 当前公司
- 当前岗位
- 所在城市
- AI 方向标签
- 校友证据链接
- 职业资料链接
- 备注

业务约束：

- 联系方式必须有
- 年龄和性别理论上可以缺失，但前 100 个不能缺
- 这 100 人要“优中选优”

## 4. 当前最重要的数据源判断

我们当前认为，一期应该优先打通以下来源：

### 第一优先级

1. Bonjour 中显性写 `浙江大学 / 浙大 / ZJU` 的人
2. 浙江大学知名实验室、课题组、导师主页中的毕业去向和成员页

已知有价值的实验室方向包括：

- CAD&CG 国家重点实验室
- DCAD 实验室
- VIPA 实验室
- 人工智能研究所
- 工业控制研究所

### 第二优先级

3. 浙大官方新闻、学院页面、活动页面
4. 微信公众号文章
5. 个人主页、访谈页、演讲页

### 第三优先级

6. GitHub
7. LinkedIn
8. 公司官网团队页
9. 公开论文作者页

## 5. 当前技术判断

我们已有一个旧项目 `seeku`，它本质上是一个 evidence-first 的 AI 人才搜索引擎。

目前我们的判断是：

- `Seek ZJU` 应该独立起项目
- 但底层可以复用 `seeku` 的部分能力

我们认为可复用的包括：

- 多源 source profile 归一化
- person / identity merge
- evidence items 存储
- 网页抓取与 LLM 摘要
- 搜索与导出链路

我们认为必须新建的包括：

- 教育背景模型
- 联系方式模型
- 校友识别规则
- 校友置信度评分
- AI 相关度评分
- review queue
- delivery batch

## 6. 已有数据验证

我们在现有 `seeku` 本地库中做了初步扫描：

- 用 `浙江大学 / 浙大` 搜，命中的 `persons` 有 8 人
- 用 `浙江大学 / 浙大 / ZJU / Zhejiang University` 搜，命中的 `persons` 有 13 人
- 中文关键词命中的 `source_profiles` 有 15 条，全部来自 `bonjour`

这说明：

- Bonjour 确实是一个高价值的浙大显性种子源
- 但现有数据规模还远远不够支撑完整交付

## 7. 你需要重点回答的问题

请你基于以上背景，重点评审以下问题：

1. `Seek ZJU` 独立起项目这个决策是否合理？
2. 如果独立起项目，你建议的最小仓库结构和模块边界是什么？
3. 哪些模块应当直接从 `seeku` 迁移？哪些不要迁移？
4. 一期 MVP 的最小 schema 应该怎么设计？
5. `person_educations` 和 `contact_methods` 是否应该在一期就做成独立表？
6. 校友置信度、AI 相关度、联系方式有效度，最合理的规则设计是什么？
7. 微信公众号文章提取，技术上应该作为主发现源还是证据增强源？
8. 实验室毕业去向页面，应当建成独立 source，还是先作为 `web` source 的 subtype？
9. 一期 100 人交付，从工程和数据角度是否现实？最可能卡在哪？
10. 你建议的第一阶段实施顺序是什么？

## 8. 希望你输出的格式

请按下面结构回答：

### A. 总体判断

- 是否同意 `Seek ZJU` 单独起项目
- 最大的对与最大的风险分别是什么

### B. 技术方案建议

- 推荐的 repo / package 结构
- 推荐的 schema
- 推荐的数据流

### C. MVP 边界

- 一期必须做
- 一期不该做
- 哪些可先用人工兜底

### D. 风险与修正建议

- 你认为我们当前方案里最容易高估或低估的地方

### E. 具体下一步

- 请给出一个按优先级排序的前 10 条行动清单

## 9. 额外要求

请不要把这个项目理解为“给 `seeku` 换几个新数据源”。

请把它当作一个新的“校友情报产品”来评估，并明确区分：

- 可以复用的技术内核
- 需要重建的业务层
- 一期交付可行的最小闭环
